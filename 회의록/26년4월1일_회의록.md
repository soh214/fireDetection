# 4월 1일 회의 회의록

- 녹음 파일: `26년4월1일.m4a`
- 회의 길이: 약 40분 5초
- 주제: VQA/VLM 모델 선정, 데이터셋 구성, 토큰 추출 방식 및 중간발표 준비

## 1. 회의 개요

이번 회의에서는 화재 감지 프로젝트를 위해 실제로 사용해본 VQA/VLM 모델의 결과, 데이터셋 처리 방식, 토큰 추출 및 분류 기준을 논의했다. 학생 팀은 PaliGemma, Gemma3 계열 등 여러 모델을 Colab과 로컬 환경에서 테스트했으며, 모델별 응답 품질과 실행 가능성을 비교했다.

회의의 핵심은 다음 중간발표까지 “어떤 모델과 프롬프트를 왜 선택했는지”, “데이터셋을 어떤 기준으로 나눌지”, “토큰을 어떤 형식으로 저장하고 분석할지”를 명확히 정리하는 것이다.

## 2. 주요 논의 내용

### 모델 테스트 결과

- PaliGemma는 비교적 가벼운 모델이지만, 질문에 대한 답변이 부정확하거나 이상하게 나오는 경우가 있었다.
- 더 무거운 모델을 사용하면 답변 품질은 나아졌지만, GPU/Colab 자원 부담이 커졌다.
- 3B~4B 정도의 모델은 Colab에서 이미지당 약 2초 내외로 돌릴 수 있는 수준으로 언급되었다.
- Qwen 계열 멀티모달 모델이나 InternVL 계열 모델도 후보로 검토해볼 만하다는 피드백이 있었다.
- 반드시 “VQA 전용 모델”만 고집할 필요는 없고, 멀티모달 모델 중 프로젝트 목적에 맞는 모델을 골라도 된다는 방향으로 논의되었다.

### CLIP 활용 방식

- CLIP은 일반적인 질의응답 모델처럼 query를 넣어 답을 받는 방식이라기보다, 이미지 벡터와 텍스트 벡터의 유사도를 비교하는 모델로 이해해야 한다는 논의가 있었다.
- 따라서 CLIP에 질문을 직접 던지는 구조보다는, 여러 caption 또는 text prompt를 준비해두고 이미지와 가장 유사한 쪽을 선택하는 방식이 더 적절할 수 있다.
- 예시로 세 가지 caption 그룹을 두는 방식이 제안되었다.
  - 명확한 화재 상황
  - 불이 보이지만 실제 화재는 아닐 수 있는 애매한 상황: 요리, 캠프파이어, 폭죽 등
  - 완전히 화재가 아닌 일반 상황
- 이 분류 체계가 정해져야 이후 토큰을 어떤 기준으로 추출하고 분류할지도 결정할 수 있다.

### 데이터셋 구성

- AI Hub 데이터셋은 용량이 약 1TB 수준으로 매우 크고, 압축 해제와 파일 이동만으로도 시간이 많이 걸린다.
- 데이터는 장소별 또는 상황별 폴더로 나뉘어 있으며, 화재, 연기, 정상/대조군 등의 분류가 포함되어 있는 것으로 파악되었다.
- 비디오에서 추출된 프레임이 많아 거의 동일한 이미지가 여러 장 반복되는 문제가 있다.
- 전체 데이터를 모두 처리하기보다는 중복 프레임을 줄이고 대표 이미지를 뽑아 사용하는 방향이 현실적이다.
- 유사한 이미지가 연속으로 반복되는 경우, 일정 간격마다 한 장씩 뽑거나 픽셀 차이/간단한 알고리즘으로 중복을 줄이는 방법이 언급되었다.
- AI 모델로 중복 분류를 자동화하는 것은 데이터 용량 때문에 부담이 크므로, 우선 수작업 또는 단순 알고리즘 기반 샘플링을 고려한다.

### 데이터 그룹과 샘플 수

- 데이터는 최소 세 그룹으로 나누는 방향이 논의되었다.
  - 실제 위험한 화재
  - 불 또는 연기가 있지만 화재로 판단하면 안 되는 애매한 상황
  - 완전히 화재가 아닌 정상 상황
- 가능하다면 그룹당 약 1,000장 이상을 확보하면 통계적으로 설명하기 좋다는 의견이 있었다.
- 다만 현실적으로는 처리 가능한 만큼 확보하되, 특정 그룹에 과도하게 편향되지 않도록 균형을 맞추는 것이 중요하다고 정리되었다.
- AI Hub 데이터에는 소방관이 불을 끄는 장면, 진화 후 연기만 나는 장면, 정상 상황 등이 포함되어 있는 것으로 보인다.
- 요리, 캠프파이어, 폭죽, 흡연 등 false positive를 검증하기 위한 데이터는 별도로 수집해야 할 가능성이 높다.

### 데이터 저장 형식

- VQA/VLM 결과를 저장하는 형식으로 Excel, CSV, JSON이 논의되었다.
- JSON은 구조화에는 좋지만 데이터가 많을 때 파일 수와 저장 공간 문제가 생길 수 있다.
- CSV는 가볍고 후처리하기 쉬우므로 결과 저장 형식 후보로 제안되었다.
- 기본적으로는 이미지 이름, 모델 답변, 사용 프롬프트, 분류 라벨 등을 저장하고, 이후 단어 빈도와 토큰 통계를 내는 방식이 적절하다.

### 토큰 추출 및 분석

- VQA/VLM 모델의 답변에서 모든 단어를 그대로 쓰는 것이 아니라, 화재 판단에 유의미한 단어만 추출해야 한다.
- 예를 들어 `There is a spreading fire in the building`이라는 답변에서 `building`보다 `spreading`, `fire`와 같은 단어가 더 중요하다는 의견이 있었다.
- 답변 문장에서 불필요한 단어를 제거하고, 빈도수를 계산해 유의미한 토큰을 뽑는 방식이 제안되었다.
- 추출된 토큰은 실제 화재를 가리키는 positive token과 화재가 아님을 시사하는 negative token으로 나누는 방향이 논의되었다.

### 위성/산불 데이터셋

- 소장님이 공유한 데이터셋 목록 중 위성사진 또는 산불 관련 데이터가 포함되어 있었던 것으로 언급되었다.
- 위성사진에서는 구름과 연기 구분이 어렵고, 육안으로도 판단이 어려워 현재 프로젝트 범위에서 바로 활용하기에는 부담이 크다는 의견이 있었다.
- 장기적으로는 산불 탐지나 원거리 산림 감지로 확장할 수 있지만, 현재는 CCTV/일반 이미지 기반 화재 판단에 집중하는 것이 현실적이라는 방향으로 정리되었다.

### 컴퓨팅 자원

- Colab 사용량 제한과 GPU 부족 문제가 있었다.
- 종합설계 수업에서 클라우드/GPU 자원 지원이 가능한지 확인해보기로 했다.
- 수업 차원의 지원이 어렵다면 연구실 서버에 컨테이너를 하나 만들어 제공할 수 있다는 의견이 있었다.

### 중간발표 준비

- 원래 4월 21일경으로 예상했던 첫 중간발표가 더 앞당겨져 다음 주 화요일에 진행될 가능성이 언급되었다.
- 발표에서는 아직 최종 구현이 완성되지 않았더라도, 다음 내용을 제시해야 한다.
  - 테스트한 모델 목록과 각 모델의 문제점
  - 최종 후보 모델을 선택한 이유
  - 사용하려는 프롬프트 후보와 선택 근거
  - 데이터셋을 세 그룹으로 나누는 기준
  - 중복 데이터 처리 및 샘플링 방식
  - 토큰 추출과 빈도 분석 계획
  - 현재 한계: 데이터 용량, GPU 자원, false positive 데이터 부족 등

## 3. 결정 사항

- PaliGemma는 가볍지만 답변 품질 문제가 있어 다른 후보 모델도 검토한다.
- Qwen/InternVL 등 3B~4B급 멀티모달 모델을 추가로 테스트한다.
- CLIP은 직접 질의응답보다는 caption/text prompt 유사도 비교 방식으로 활용하는 방향을 검토한다.
- 데이터는 실제 화재, 애매한 비화재 상황, 완전 정상 상황의 세 그룹으로 나누어 구성한다.
- 전체 1TB 데이터를 모두 처리하기보다는 대표 이미지 샘플링과 중복 제거를 우선한다.
- 결과 저장 형식은 JSON보다 CSV가 현실적일 수 있으며, 최종 분석 편의성을 기준으로 선택한다.
- Colab 자원이 부족하면 수업 지원 또는 연구실 서버 컨테이너 제공 가능성을 확인한다.
- 다음 중간발표에서는 모델/프롬프트/데이터셋 선택 근거를 중심으로 발표한다.
